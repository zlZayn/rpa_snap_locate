# 演进讨论存档

本文档记录对 RPA Snap Locate 项目的整体理解、已确认边界、Command Series 梳理，以及后续功能增减的讨论思路。各节明确区分已实现、已决定和暂不实施事项，不能把存档中的远期设想直接当作开发任务。

## 1. 项目现状

### 1.1 定位

Windows 上的轻量鼠标操作录制与回放工具，核心承诺是"录一次，以后可重复执行"。项目气质为聚焦、克制、老实：聚焦在鼠标操作与时间的精确记录还原；克制在不堆功能；老实在于出错即报错、不假装成功。

### 1.2 核心能力

- 精确时间线：记录每次按下/抬起及事件间间隔，保留快速双击节奏；
- 多按键支持：左键、右键、中键；
- 屏幕缩放适应：录制时归一化坐标，回放时按当前 DPI 还原；
- 中途截图：F3 框选区域，回放时自动截取保存；
- 完整证据链：每次回放产出前后整屏快照、逐事件前后快照（红叉标点）、时间报告；
- 简化模式：legacy 模式只保存点击位置和顺序，不记节奏。

### 1.3 架构分层

项目以分层方式组织，依赖方向主要由上向下：

- 入口（main.py）：命令分发、热键注册；
- 引擎层（engine/）：状态机、事件采集、校验、调度、回放；
- 核心能力（core/）：屏幕感知、坐标定位、Windows 输入；
- 数据层（data/）：工作流持久化；
- 配置与工具（config/、utils/）：YAML 配置、无状态工具。

关键设计关系：

- 录制状态机不直接监听物理鼠标，由独立采集器以最快速度抓事件并打单调时间戳；
- 工作流保存前强制过校验，不合法事件不写成有效文件；
- 回放调度采用绝对截止时间，一次迟到不污染整条时间线；
- Windows 输入把"准备窗口/移动鼠标"与"注入点击"分离，并检查所有返回值，被拒即报错。

### 1.4 两种工作流

| 对比项 | 时间线工作流（默认） | 逐步点击工作流（legacy） |
| :--- | :--- | :--- |
| 记录单位 | 每次按下/抬起 | 每个完整点击 |
| 时间信息 | 精确事件偏移 | 仅步骤顺序 |
| 双击 | 完整保留两组按下抬起与间隔 | 只能存两个点击，无法还原节奏 |
| 支持按键 | 左/右/中 | 固定左键 |
| 文件标记 | version 5.0 | version 4.0 或无标记 |
| 适合场景 | 需精确还原节奏 | 只需点对位置 |

### 1.5 时间模型

- 双时钟：created_at 墙上时钟给人看，offset_ns 单调时钟给程序排序回放；
- 时间原点：每次按 F2 开始或继续录制时建立，首个操作前的等待也会被记录；
- 分段规则：F2 停止后的暂停时间不记录，再次按 F2 后建立新的分段时间原点，保存时只拼接各分段内部时间；
- 整数纳秒存储：避免浮点连加的累积误差；
- 纳秒仅为存储单位：实际精度受 Windows 钩子与 Python 线程调度限制，非硬实时。

### 1.6 当前局限

- 只录鼠标按键，不录键盘、滚轮、移动轨迹；键盘自动化通过 Command Series 调用外部工具完成；
- 拖拽明确不支持：不记录移动轨迹，up 也不存独立坐标，无法可靠还原；
- 内置 `LLMLocator` 空壳已删除；界面变化检测、图像特征工具仍为预留未实现；
- timing_mode、evidence_mode 已声明但未驱动分支；box_select_timeout_seconds 已被读取，但当前没有定时器执行超时取消；
- Windows/Python 非硬实时环境，严格调度减少漂移但不保证零误差。

## 2. Command Series 梳理

### 2.1 核心论点

PowerShell/Bash 脚本本身就是最好的命令编排器，RPA 回放只是脚本里的一个普通前台步骤，不造新框架。这决定了整个方案轻、透明、好调试的气质。

### 2.2 步骤的两种身份

- 前台命令：脚本等它跑完、看退出码再走下一步，典型如 RPA 回放、Python 脚本、测试；
- 后台进程：启动后脚本立即继续，需要时显式等，典型如 GUI 软件、长期服务。

### 2.3 五个可组合原子脚本

| 原子 | 作用 | 关键细节 |
| :--- | :--- | :--- |
| `atoms/run-command.ps1` | 跑任意前台命令 | 检查命令存在、执行并检查退出码 |
| `atoms/start-app.ps1` | 后台启动 GUI | `.cmd`/`.bat` 启动时隐藏控制台宿主 |
| `atoms/wait.ps1` | 固定等待 | 没有明确就绪检测时使用 |
| `atoms/run-rpa.ps1` | 回放 RPA 工作流 | 补全路径、校验文件存在、检查退出码 |
| `atoms/keyboard.ps1` | 键盘输入或快捷键 | 先激活指定窗口，失败时不发送按键 |

### 2.4 错误处理铁律

1. GUI 启动失败立即停；
2. RPA 非零退出码立即停并输出工作流路径；
3. 可恢复步骤才显式重试；
4. 脚本先切到项目根目录；
5. 每步输出稳定名称日志。

### 2.5 Windows 权限

UIPI 机制导致低权限进程不能向高权限窗口注入鼠标事件。规则：目标软件与回放命令必须权限对齐。通用模板不自动提权，具体 Series 可在启动子进程前自举提权（见 positron-open-folder.ps1）。

### 2.6 独立 runner 的触发条件

文档明确延迟复杂度：出现统一重试/超时/结构化日志、动态拼装步骤、跨 shell 共用流程、暂停恢复可视化等需求时，才考虑造独立 runner。在此之前，原生脚本是最轻最透明的 Series runner。

## 3. 后续功能增减的判断原则

### 3.1 四个筛选问题

1. 是否触及"录制/回放"核心职责？
2. 是否必须访问项目内部状态（事件队列、单调时钟、v5 格式）？
3. 能不能由外部脚本/CLI 拼装完成？
4. 会不会让项目从"录放工具"变成"自动化大杂烩"？

核心立场：编排不属于内核，交给脚本。

### 3.2 该做减法的

- 已删除 `LLMLocator` 空壳及其工厂注册，避免把不可用的 `llm` 定位方式伪装成受支持能力；
- timing_mode、evidence_mode 和 box_select_timeout_seconds 的去留尚未决定，不在本轮顺带删除；其中框选超时配置已被读取，需单独核查行为后再处理。

### 3.3 该补强承诺的

- 异常补发抬起（Scheduler 的 finally 逻辑）补测试覆盖与明确日志，因为鼠标卡在按住状态是体验最差的故障；
- 拖拽已明确不支持并写入 README；不扩展按下抬起配对模型。

### 3.4 暂不实施的增值想法

- 录制后只读查看能力当前不做。未来如果确有需求，可由独立脚本读取 JSON，不必增加内核子命令。

### 3.5 该外部化的

| 项 | 理由 |
| :--- | :--- |
| 批量回放、定时回放 | Series + 系统 Task Scheduler/cron 即可 |
| 回放结果断言/对比 | 断言逻辑千差万别，项目只产证据不下结论 |
| 回放产物归档与清理 | 清理策略因人而异，外部脚本扫目录即可 |
| 工作流列表与管理 | ls 或十行脚本即可 |
| 跨机器分发、版本管理 | git 管工作流目录，文件同步分发 |
| 回放报告聚合统计 | 纯外部数据处理，项目只负责产标准 JSON |

### 3.6 已决定的协作边界

- 键盘录制：不加入本项目。键盘输入与快捷键由 `examples/atoms/keyboard.ps1` 作为 Series 的独立原子执行，也可替换为其他具有明确退出码的外部键盘工具；
- GUI：强烈不建议，引入框架/打包负担，录制时干扰操作，破坏克制气质。

## 4. 视觉大模型结合思路

本节为远期边界存档，当前不实施。已确认本项目不内置大模型调用；未来如有需求，视觉大模型能力放在另一个项目，两者通过 Series 串联协作。

### 4.1 作者的判断

- 本项目不内置大模型调用，保持不冗余；
- 大模型能力放在另一个项目里；
- 两个项目通过 series 串联协作。

这个架构比"内置 LLM 定位"更优，原因见 4.3。

### 4.2 F3 截图是天然的接口

本项目已具备的能力：录制时按两次 F3 框一个矩形，该区域被写进时间线，回放到那个时刻自动把这块区域截下来，存成 screenshots/event_N.png。

也就是说，本项目已经能在回放的精确时间点，产出一块"你关心的屏幕"的图片。这张图片就是喂给视觉大模型的完美输入。所以"截图区域 + 视觉大模型"的数据通路已经通了，不需要改本项目。本项目只管按时截好图存好，视觉模型只管看图判断。

### 4.3 为什么不内置是对的

| 对比项 | 内置 LLM 定位 | 外部项目 + series 串联 |
| :--- | :--- | :--- |
| 本项目复杂度 | 增加模型调用、超时、重试逻辑 | 零增加 |
| 时间线精度 | 模型延迟几百毫秒到几秒，破坏节奏 | 截图在回放时已产出，模型离线慢慢看，不影响时间线 |
| 模型选择 | 锁死在代码里 | 换模型只改另一个项目 |
| 失败处理 | 和回放绑在一起，难调试 | 各自独立调试 |
| 可替换性 | 难 | 视觉项目可单独换、单独升级 |

最关键的一点：内置 LLM 定位会拖慢回放（要等模型返回才能继续点），而本项目卖点是精确时间线。把模型挪到外面，回放该多快多快，图截完了慢慢分析，两者解耦。

### 4.4 串联的典型形态

series 脚本大致形态：

1. 调用本项目回放工作流，结束后 screenshots/ 下已产出 event_N.png；
2. 调用另一个项目（视觉大模型）读图判断，传入图片路径与问题，返回退出码与可选 JSON 报告；
3. 根据退出码决定后续动作（失败则抛错停止，成功则继续）；
4. 继续后续流程（归档、通知等）。

本项目产出图片，视觉项目消费图片，series 负责按顺序接起来。三者各管一摊，互不知晓对方内部实现。

### 4.5 视觉大模型项目的职责边界

该承担的：

- 接收图片路径（或目录）；
- 带着问题调用视觉大模型（本地模型或 API，属其内部事务）；
- 返回结构化结果（退出码 + 可选 JSON 报告）。

不该承担的：

- 不该知道 RPA 工作流格式；
- 不该依赖本项目任何代码；
- 不该负责截图；
- 不该负责编排。

这样该视觉项目本身也是小、专、可独立替换，与本项目气质一致。

### 4.6 两种用法的区分（远期存档，当前不做）

以下内容仅用于解释边界，不进入当前行动计划。

#### 用法 A：回放后离线校验

回放已经跑完，图已截好，视觉模型事后看图判断"这次操作对不对"。不影响时间线，推荐先做。串联方式见 4.4。

#### 用法 B：回放中实时决策

回放到某个点，截一张图，立刻问视觉模型"现在该点哪里"，再根据回答决定下一步点击。这种用法必然破坏时间线（因为要等模型），本质上已经不是"录放"，而是"智能自动化"。

#### 已确认的当前边界

- 当前两种用法都不实施；
- 未来若做用法 A，通过 Series 串联外部视觉项目，不修改本项目内核；
- 用法 B 不属于本项目的精确录放职责，如未来探索，应作为独立项目重新设计。

本项目不为远期设想保留不可用的 `LLMLocator` 空壳。

## 5. 行动优先级

1. 已完成：删除 `LLMLocator` 空壳，明确拖拽不支持；
2. 待核查：异常补发抬起的测试覆盖与日志是否充分；
3. 待单独决定：未生效配置项的删除或实现，不与其他工作捆绑；
4. 持续遵守：键盘、批量、定时、断言、清理、统计、分发均通过 Series 与外部工具协作；
5. 暂不实施：视觉大模型、录制查看工具及其他增值功能。

## 6. 总原则

项目守住"精确录制 + 精确回放 + 产出证据"这一条主线，其余一切（编排、断言、清理、分发、统计、智能识别）全部交给 Command Series 脚本和外部工具。

目标是让项目长期保持小、准、老实的气质，复杂度涨在脚本层而不是内核层。

## 7. 仍待讨论的问题

- timing_mode、evidence_mode 是否直接删除；
- box_select_timeout_seconds 是补上真正的超时取消，还是连同读取逻辑一起删除；
- 异常补发 mouse_up 的现有测试和日志是否已经足够，不足时只补可靠性测试与日志。

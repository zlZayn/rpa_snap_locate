# 项目技术设计文档：轻量级视觉驱动自动化框架 (未定档)（已过时）

**版本**：V2.0（增强结构版）  
**核心语言**：Python
**设计哲学**：**简单而不简陋，精悍而不笨重；组件可插拔，算法靠数学，坐标靠物理映射。**

---

## 一、项目目标与边界

### 1.1 核心目标

构建一个**纯视觉驱动（VLM/传统CV混合）**的桌面/Web自动化工具。它通过“看”屏幕截图理解界面，并模拟鼠标键盘操作，完成预设的线性流程任务。

### 1.2 明确的能力边界（Do's & Don'ts）

| 维度 | 能做到（✅） | 不涉及（❌） |
| :--- | :--- | :--- |
| **操作对象** | Windows/macOS 任何图形界面（Win32, Qt, Web, Java Swing）。 | 命令行终端（纯字符）、无UI后台服务、移动端触屏手势。 |
| **操作类型** | 左/右键点击、文本输入、组合键、滚动、悬停。 | 拖拽上传文件、双指缩放、3D空间操作。 |
| **流程复杂度** | 预设任务链（≤10步），单步原子回滚，异常隔离。 | 实时自主长流程规划（非Agent框架，无自我反思循环）。 |
| **成本控制** | 内置dHash缓存机制，节省70%无效大模型API调用。 | 完全离线运行（必须依赖云端或本地部署的多模态模型）。 |

## 二、整体架构设计（三层分离 + 双环容错）

框架采用 **“感知-决策-执行”** 三层解耦，并引入 **“外环（流程调度）”** 与 **“内环（原子动作）”** 双环容错机制。

```text
[ 输入: task_sequence.json ] 
         ↓
┌─────────────────────────────────────────────────────────────┐
│                  外环 (Pipeline 调度器)                      │
│  - 顺序遍历任务列表                                         │
│  - 异常捕获与隔离（某步失败不中断整体）                     │
│  - 任务级重试（指数退避）                                  │
└─────────────────────────────────────────────────────────────┘
         ↓ (逐项分发)
┌─────────────────────────────────────────────────────────────┐
│           内环 (原子动作执行器 - Atomic Actor)               │
│  ┌───────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐ │
│  │ 感知模块  │→│ 定位模块 │→│ 执行模块 │→│ 验证模块 │ │
│  │ (截图/dHash)│  │(Locator)│  │(Action) │  │(dHash比 │ │
│  └───────────┘  └──────────┘  └──────────┘  │  对/LLM)│ │
│                                              └──────────┘ │
└─────────────────────────────────────────────────────────────┘
```

## 三、组件模块详细设计（组件化核心）

为满足“先拆掉LLM跑固定点击”的需求，所有依赖均面向接口编程。

### 3.1 感知模块 (`Perception`)

- **职责**：屏幕捕获、图像预处理、图像指纹计算。
- **关键技术**：
  - 使用 `mss` 进行高速截屏（物理像素）。
  - 实现 **dHash（差异哈希）** 算法，将截图压缩为整数哈希值，用于极速界面变化比对。
  - 将PIL图像转为Base64，供未来LLM模块调用。

### 3.2 定位模块 (`Locator`) —— **可插拔核心接口**

- **抽象基类**：`BaseLocator`，定义 `locate(step_config) -> (x, y)`。
- **实现A：`FixedLocator`**（当前阶段）  
  直接从JSON配置文件中读取预设的归一化坐标或物理坐标，零成本运行。
- **实现B：`LLMLocator`**（未来阶段）  
  接收截图Base64和Prompt，调用多模态大模型API，解析返回的JSON坐标。切换时只需修改一行工厂代码，对其他模块零侵入。

### 3.3 执行模块 (`Action`)

- **职责**：封装 `pyautogui` 底层指令。
- **核心算法**：
  - **坐标边界裁剪算法**：将超出屏幕范围（如负数或大于分辨率）的坐标自动拉回边界内，防止模型幻觉导致鼠标飞至左上角触发紧急停止。
  - **双坐标空间映射器**：内置DPI缩放比（`scale = 物理尺寸/逻辑尺寸`）。执行时传入的是逻辑坐标，确保在不同缩放比屏幕上的点击准确性。

### 3.4 验证与缓存模块 (`Validator`)

- **核心算法：低成本变化预检**  
  在执行点击前计算截图dHash，点击后轮询计算新截图dHash。
  - 若哈希不同（界面变了），才允许进入下一步或调用昂贵的LLM做语义验证；
  - 若哈希相同（界面卡死/未响应），直接触发内环重试，**完全不消耗API Token**。

### 3.5 坐标增强模块（解决“窗口漂移”与“缩放适配”）

这是V2.0增强结构的核心，包含三个子算法：

1. **窗口锚点吸附算法**：
    - 录制时，记录目标相对于**当前活动窗口左上角**的归一化偏移 `(norm_x, norm_y)`。
    - 回放时，先通过系统API（`pygetwindow`）获取窗口实时左上角坐标，再计算 `物理坐标 = 窗口左上角 + (norm_x * 窗口宽度, norm_y * 窗口高度)`。这解决了窗口在屏幕上拖拽移动导致坐标失效的问题。

2. **特征锚点模板匹配算法（进阶）**：
    - 录制时，在目标按钮周围自动截取一块**高信息熵（纹理丰富）**的小图作为“锚点模板”，并计算锚点中心到目标中心的相对偏移量。
    - 回放时，使用 `cv2.matchTemplate` 在截图中搜索该锚点模板。
    - **降级策略**：若匹配度低于阈值（如0.8），自动在匹配点附近±30像素区域精细扫描，或对图像进行缩放金字塔匹配，大幅提高复杂背景下的定位鲁棒性。

3. **DPI自适应映射算法**：
    - 录制时记录当前系统的DPI缩放比（如2.0）。
    - 存储坐标时统一换算为 **“逻辑归一化坐标”（0~1浮点数）**，基于录制时的逻辑分辨率（如1920x1080）计算。
    - 回放时，`pyautogui` 接收的坐标 = `逻辑归一化坐标 * 当前系统的逻辑分辨率`。这确保了代码在 2K、4K 屏幕间无缝迁移。

## 四、核心工作流与数据流（串联逻辑）

系统运行时，遵循以下严格时序：

### 阶段一：录制（离线配置）

1. **交互**：开发者将鼠标悬停于目标元素，按下热键（如F2）。
2. **采集**：脚本捕获当前鼠标物理坐标、活动窗口句柄与DPI缩放。
3. **计算**：将物理坐标映射为逻辑归一化坐标。
4. **增强（可选）**：在目标周围截取特征锚点图，并保存至本地缓存目录。
5. **持久化**：将步骤、坐标、锚点图路径、动作类型写入 `tasks/workflow.json`。

### 阶段二：回放（在线执行）

1. **加载**：引擎读取JSON，根据配置实例化对应的 `Locator`（当前为Fixed）。
2. **外环迭代**：`for step in steps`。
3. **内环原子操作（单步）**：

- **感知**：截取当前屏幕（物理图）。
- **定位**：调用 `locator.locate()`（此时若为固定模式，直接读配置；若为未来LLM模式，则传图给API）。
- **执行前缓存**：计算当前屏幕dHash，存入 `before_hash`。
- **执行**：`Action` 模块接收坐标，经DPI映射器换算后执行点击/输入。
- **异步等待与验证**：调用 `wait_for_change(before_hash)`，以指数退避间隔（0.1s, 0.2s, 0.4s...）轮询屏幕哈希。若超时（3s）仍无变化，判定动作失败。
- **重试机制**：内环失败则立即重试（最多3次），每次重试重新执行定位（捕获新的窗口位置/锚点）。
- **状态回写**：若内环彻底失败，记录失败信息，但**外环继续执行下一个任务**（异常隔离）。

## 六、关键技术指标与容错参数（预设）

| 参数项 | 推荐值/策略 | 备注 |
| :--- | :--- | :--- |
| **单步超时** | 15秒（含模型响应） | 超时触发外环重试 |
| **内环重试次数** | 3次 | 每次重试重新截图定位 |
| **外环任务隔离** | 是（失败跳过） | 避免单点故障阻塞整体 |
| **dHash轮询间隔** | 指数退避 (0.1, 0.2, 0.4...) | 动态适应界面响应速度 |
| **锚点模板匹配阈值** | 0.75（可配置） | 低于阈值触发二次精细扫描 |
| **边界裁剪安全区** | ±10像素 | 防止鼠标滑出屏幕边界 |

## 七、未来扩展：接入LLM的“插头”方案

当需要从“固定点击”升级为“语义理解点击”时，仅需两步操作：

1. **实现子类**：创建 `LLMLocator(BaseLocator)`，重写 `locate` 方法。内部逻辑：截全屏→转Base64→构造Prompt（“返回JSON格式坐标”）→调用API→解析返回。
2. **修改工厂**：将 `runner.py` 中的 `self.locator = FixedLocator()` 替换为 `self.locator = LLMLocator(api_key)`。

**上下游无需改动**：Perception仍负责截图，Action仍负责执行，Validator仍负责dHash缓存预检（若界面无变化，甚至可以不调用LLM直接重试）。

## 八、总结

此方案终稿具备以下特质：

1. **工程解耦性**：6大核心模块职责单一，通过接口交互，满足“先跑固定逻辑，后插AI大脑”的渐进式开发需求。
2. **算法精妙性**：以dHash替代截图比对节省算力，以双环（内/外）重试保障执行韧性，以物理归一化和锚点匹配对抗坐标漂移。
3. **生产就绪性**：涵盖了DPI缩放、窗口吸附、边界安全钳、指数退避等工业级细节，而非仅停留在Demo层面。
4. **成本可控性**：缓存机制确保在界面加载、动画播放等“无效帧”期间不请求昂贵的大模型服务。
